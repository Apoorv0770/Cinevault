1.1 Product Overview 
CineVault is a full-stack, database-driven web application designed to function as a comprehensive 
entertainment information platform — combining the functionality of IMDb's movie catalogue with the 
discovery and watchlist features of modern OTT platforms like Netflix and Hotstar. Built using Python 
Flask, MySQL, and Vanilla JavaScript, CineVault delivers a seamless, data-rich experience for movie 
discovery, user reviews, watchlist management, and intelligent recommendations. 
 
1.2 Problem Statement 
The entertainment landscape is fragmented. Users must visit multiple platforms to find movie information, 
read community reviews, manage their watch queues, and discover new content. Existing platforms like 
IMDb are information-heavy but do not provide OTT-style watchlist tracking with status management. 
Meanwhile, OTT apps focus on their own catalogue. CineVault bridges this gap by consolidating these 
features into a single, open platform with no catalogue restrictions. 
 
Specifically, CineVault solves the following problems: 
• No single open platform combines structured movie data, community reviews, watchlist tracking, 
and ML-based personalization. 
• Cold-start users receive generic recommendations; CineVault's smart routing personalizes 
recommendations progressively as users engage. 
• Community review systems on existing platforms lack social features such as review upvotes, 
user profiles, and review history. 
• Trending and box-office analytics are rarely surfaced in a unified interface alongside catalogue 
browsing. 
 
1.3 Target Users 
CineVault is designed for three primary user segments: 
• Movie enthusiasts who actively track their watch history, rate films, and seek intelligent 
recommendations. 
• Students and researchers who need structured, queryable movie data with ratings, genres, 
directors, and box office figures. 
• Casual viewers who want a clean, ad-free interface to browse trending films and build a 
watchlist without signing up for multiple OTT subscriptions. 
 
1.4 Key Differentiators 
• Adaptive ML recommendation engine that intelligently switches between three strategies — 
content-based, collaborative, and cold-start — based on user engagement data. 
• Unified platform covering discovery, watchlist status tracking (Want to Watch / Watching / 
Completed), reviews, and analytics.

"To be the most complete, community-driven movie intelligence platform — where every user, regardless 
of their engagement level, receives a personalized, data-rich entertainment discovery experience."

genre, year, IMDb score, and release year.

and recommendation updates.

3. Features & Functional Requirements 
 
3.1 Authentication System 
3.1.1 User Registration 
• Accepts: username (3-50 characters, alphanumeric + underscore), email (RFC 5322 format), 
password (minimum 6 characters). 
• Server-side validation rejects duplicate usernames and email addresses with distinct HTTP 409 
Conflict responses. 
• Passwords are hashed using bcrypt with a randomly generated salt before storage — raw 
passwords are never persisted. 
• On successful registration, the user is automatically logged in and a server-side session is 
created. 
 
3.1.2 Login & Session Management 
• Email-based login with bcrypt password verification. Invalid credentials return a generic error to 
prevent user enumeration. 
• Sessions are stored server-side using Flask's signed cookie mechanism. The session cookie is 
HttpOnly and SameSite=Lax for XSS and CSRF protection. 
• GET /api/auth/me enables the frontend to check authentication state on page load without 
requiring re-login. 
 
3.1.3 User Profile 
• Users can update their bio (max 500 characters) and avatar URL. Public profiles display review 
count, watchlist count, and average rating given. 
• Password change requires verification of the current password before the new hash is stored. 
 
3.2 Movie Browsing & Discovery 
• The movie catalogue displays all films with: title, release year, PG rating, duration, IMDb score, 
Metascore, genres, director, and poster image.

• Full-text search across four fields: title, plot synopsis, director name, and cast list. 
• Results are ranked with title matches surfaced first, followed by relevance by IMDb score. 
• Minimum query length of 2 characters enforced server-side. Results capped at 20 for 
performance. 
• Search is performed using SQL LIKE queries with parameterized inputs — no raw string 
interpolation. 
 
3.4 Reviews & Ratings 
• Users submit reviews with a numeric rating (1-10 scale) and an optional text comment (max 
2,000 characters). 
• One review per user per movie is enforced at the database level using a UNIQUE constraint 
(UNIQUE KEY one_review_per_user). Submitting again updates the existing review (ON 
DUPLICATE KEY UPDATE). 
• Reviews are displayed in a paginated feed sorted by likes count (descending), then by recency. 
Pagination is controlled via ?page= and ?per_page= parameters. 
• Each movie's review page shows the aggregate average rating and total review count alongside 
individual reviews. 
• Review likes: users can toggle a like on any review. The system uses a review_likes junction 
table to prevent double-liking. The likes_count on the review row is maintained denormalized for 
fast display. 
• Users can delete their own reviews. Review ownership is enforced server-side by matching 
session user_id. 
 
3.5 Watchlist Management 
• Users can add any movie to their personal watchlist with an initial status of want_to_watch. 
• Three status states are supported: Want to Watch, Watching, and Completed — tracked via a 
MySQL ENUM column. 
• Status can be updated at any time via PUT /api/watchlist/status/<movie_id>, enabling a 
Kanban-style progress tracker. 
• Watchlist supports filtering by status (?status=completed) to display curated views.

3.6 Recommendation System 
The recommendation engine is the most technically sophisticated component of CineVault. It implements 
a smart routing architecture that selects between three distinct recommendation strategies based on the 
user's engagement profile. Detailed technical treatment appears in Section 9. 
• Strategy 1 — Genre-Based (Content Filtering): Triggered when watchlist has 3+ entries. 
• Strategy 2 — Collaborative Filtering: Triggered when review count has 3+ entries but watchlist 
is sparse. 
• Strategy 3 — Top-Rated (Cold Start): Fallback for new or unauthenticated users. 
• Supplementary — Similar Movies: Genre-overlap similarity for a specific source movie (More 
Like This). 
• Supplementary — By Director: All films by the same director, ordered by IMDb score. 
 
3.7 Trending & Analytics 
• Trending scores are computed dynamically: each GET /api/movies/<id> call increments both 
view_count and score in the trending table using an upsert (INSERT ... ON DUPLICATE KEY 
UPDATE). 
• Top 10 trending movies are exposed via GET /api/movies/trending, joining the trending table 
with movie details and genres. 
• Box office rankings are stored in the box_office table with week_of date, rank, and gross — 
accessible via GET /api/movies/boxoffice. 
• Editorial content (director spotlights, upcoming releases, award season coverage) is managed 
in the editorials table and surfaced via GET /api/movies/editorials.

6. System Architecture 
 
6.1 Architecture Overview 
CineVault follows a classic three-tier web application architecture: a static frontend tier, a RESTful API 
middleware tier (Flask), and a relational data persistence tier (MySQL). The ML recommendation logic is 
embedded within the Flask middleware tier as a set of SQL-driven algorithms, not as a separate 
microservice. 
 
6.2 Frontend Tier 
• Technology: HTML5, CSS3 (with Tailwind-style utility classes), Vanilla JavaScript (ES6+). 
• Structure: Two primary HTML files served by Flask as static assets — index.html 
(home/catalogue) and movie.html (movie detail page). All routing is handled server-side via 
Flask's @app.route decorators. 
• API Communication: All data fetching uses the browser's native fetch() API with async/await 
syntax. The frontend is decoupled from the backend — it only receives JSON and renders it 
dynamically into the DOM. 
• Session State: The frontend calls GET /api/auth/me on load to determine authentication state. 
UI elements (Add to Watchlist, Review Form, Recommendations) are conditionally rendered 
based on login status. 
• No frontend framework dependency: the choice of Vanilla JS eliminates build toolchain 
complexity, making the project easy to run locally without Node.js, webpack, or npm. 
 
6.3 Backend Tier — Flask Application 
• Application Factory Pattern: create_app() in app.py initializes all extensions and registers 
blueprints. This pattern enables environment-specific configuration and simplifies unit testing. 
• Blueprint Architecture: the application is organized into five domain-specific blueprints — auth, 
movies, reviews, watchlist, and recommendations — each prefixed with /api/<domain>. This 
enforces separation of concerns and makes the codebase navigable. 
• Extensions: Flask-MySQLdb provides MySQL connectivity; Flask-CORS handles cross-origin 
headers (configured to localhost:5000). Extensions are initialized in extensions.py to avoid 
circular imports — a standard Flask best practice. 
• Session Management: Flask's server-side session stores user_id and username in a signed, 
encrypted cookie. The @login_required decorator (implemented as a Python functools.wraps 
closure) protects all authenticated endpoints. 
• Input Validation: all route handlers validate input type (int/str), length, format (regex for email), 
and range (1-10 for ratings) before any database operation. Parameterized queries are used 
throughout — no string interpolation in SQL. 
 
6.4 Database Tier — MySQL

• Schema: 11 tables organized around core entities (users, movies, genres) and their

problem. 
 
6.5 Request-Response Cycle 
A full request lifecycle for GET /api/movies/?genre=Sci-Fi&sort=imdb_score&order=DESC: 
26. Browser sends HTTP GET request with query parameters. 
27. Flask's URL dispatcher matches the route to movies_bp.get_all_movies(). 
28. Route handler validates query params: checks genre is a string, year is numeric, sort is 
whitelisted, order is ASC/DESC. 
29. Parameterized SQL query executes against MySQL — joins movies, movie_genres, and genres 
tables with optional WHERE clause. 
30. MySQL returns dict rows via DictCursor → Flask serializes to JSON using jsonify(). 
31. HTTP 200 response with JSON body → JavaScript fetch() receives it → DOM is updated.

9. Machine Learning Integration 
 
9.1 Overview 
CineVault's recommendation engine implements a layered, SQL-driven machine learning pipeline that 
delivers personalized recommendations without requiring a separate ML microservice. The 
recommendation logic is embedded directly within the Flask blueprint (blueprints/recommendations.py), 
making it testable, maintainable, and deployable without additional infrastructure. 
 
The system employs three complementary strategies: content-based filtering, user-based collaborative 
filtering, and a cold-start handler — all orchestrated by an intelligent routing layer that selects the optimal 
strategy based on each user's engagement profile. 
 
9.2 Strategy 1 — Content-Based Filtering (Genre Overlap) 
Trigger Condition 
Activated when the user has 3 or more movies in their watchlist. 
Algorithm 
The algorithm extracts the set of genre IDs from all movies in the user's watchlist. It then identifies all 
movies that share at least one genre with this set, excludes movies already in the watchlist, and ranks 
results by the count of shared genres (match_score) followed by IMDb score as a tiebreaker. 
SQL Implementation 
• A correlated subquery extracts the unique genre IDs from the user's watchlist via a JOIN on 
movie_genres. 
• The outer query matches candidate movies on these genre IDs using a JOIN (not a subquery in 
the WHERE clause), enabling the COUNT(DISTINCT genre_id) aggregate to produce the 
match_score. 
• An exclusion subquery (NOT IN watchlist) ensures already-seen content is not recommended. 
Strengths & Limitations 
• Strength: works with zero cross-user data; pure preference-based. Does not suffer from data 
sparsity. 
• Limitation: does not account for the user's rating quality — a movie added to watchlist but rated 
2/10 still influences genre weights. 
 
9.3 Strategy 2 — User-Based Collaborative Filtering

CineVault v2.0  |  Final Year Project  |  Page 16 of 24  Trigger Condition 
Activated when the user has 3 or more reviews submitted but fewer than 3 watchlist entries. 
Algorithm 
This strategy identifies similar users (users who have rated the same movies with ratings within 2 points 
of the current user's ratings) and then recommends movies that those similar users have rated 7 or higher 
which the current user has not yet reviewed. 
SQL Implementation 
• Self-join on the reviews table (r1 = current user, r2 = other users) where r1.movie_id = 
r2.movie_id AND ABS(r1.rating - r2.rating) <= 2 identifies rating-similar users. 
• A second join (r3) retrieves the highly-rated movies (rating >= 7) from those similar users. 
• Results are aggregated by community_score (average rating from similar users) and 
similar_users_count, giving confidence weighting. 
• HAVING similar_users_count >= 1 ensures at least one corroborating similar user. 
Strengths & Limitations 
• Strength: discovers serendipitous content outside the user's own genre preferences — the 
hallmark of collaborative filtering. 
• Limitation: subject to cold-start for new users and sparse data; requires the platform to 
accumulate review data across users. 
 
9.4 Strategy 3 — Cold-Start (Top-Rated Fallback) 
Trigger Condition 
Activated for all unauthenticated users and for new users with fewer than 3 watchlist entries and fewer 
than 3 reviews. 
Algorithm 
Returns the 6 movies with the highest combined score — computed as 
COALESCE(AVG(community_rating), imdb_score) — ensuring that movies with no community ratings 
fall back to their IMDb score rather than being excluded. 
This is a popularity-based recommendation strategy, a standard industry approach for handling the cold-
start problem. 
 
9.5 Smart Routing Logic 
The GET /api/recommendations/ endpoint implements a decision tree that runs two COUNT queries 
before selecting a strategy: 
32. Query watchlist count for current user. 
33. Query review count for current user. 
34. If watchlist_count >= 3: use genre-based (content filtering). 
35. Else if review_count >= 3: use collaborative filtering. 
36. Else if watchlist_count > 0: use genre-based (partial data). 
37. Else (no data or anonymous): use top-rated (cold start).

9.6 Supplementary Recommendation Modes 
Similar Movies (More Like This) 
For a given source movie, the algorithm counts shared genre IDs with every other movie in the catalogue 
and returns the top 4 by shared_genres count, with IMDb score as tiebreaker. This powers the 'More Like 
This' section on every movie detail page. 
By Director 
Retrieves all other movies by the same director, ordered by IMDb score. This surfaces a director's 
filmography, supporting cinephiles who follow auteur directors.

10.1 Performance 
• All API endpoints must respond within 300ms for catalogue queries on a local MySQL instance 
with up to 10,000 movie records. 
• The recommendations endpoint may take up to 500ms due to multi-table self-join complexity; 
this is acceptable for a non-real-time feature. 
• Movie detail pages execute 3 queries (movie data, trending upsert, view log insert) — these are 
kept sequential to ensure view count accuracy. 
• GROUP_CONCAT for genre aggregation is performed at the SQL layer to avoid N+1 query 
patterns in the Python application layer. 
 
10.2 Scalability 
• The Application Factory pattern (create_app()) enables horizontal scaling — multiple Flask 
worker processes can be spawned behind a load balancer (Gunicorn + Nginx in production). 
• The stateless JSON API design means any request can be handled by any worker — session 
state is in the client-side cookie, not in-process memory. 
• MySQL connection pooling is handled by Flask-MySQLdb's underlying MySQLdb C extension. 
• The trending system uses UPSERT semantics (INSERT ... ON DUPLICATE KEY UPDATE) to 
handle concurrent view increments without race conditions. 
 
10.3 Security 
• Authentication: bcrypt with per-password random salts. No raw passwords stored. No 
predictable hashing (MD5/SHA1 not used). 
• Session: HttpOnly cookies prevent JavaScript session theft. SameSite=Lax prevents CSRF in 
cross-origin form submissions. 
• SQL Injection: all database queries use parameterized inputs (%s placeholders with MySQLdb). 
No f-string or .format() interpolation into SQL. 
• Input Validation: sort column whitelist prevents ORDER BY injection. All integer inputs 
(movie_id, rating) are explicitly cast and validated. 
• CORS: restricted to localhost:5000 in development. In production, this would be locked to the 
application's domain. 
• Secret Key: Flask's SECRET_KEY signs session cookies. Must be changed to a 
cryptographically random value in production. 
 
10.4 Usability 
• The frontend requires no JavaScript framework installation — runs in any modern browser by 
simply opening the served HTML page. 
• All interactive actions (Add to Watchlist, Submit Review, Like) provide immediate UI feedback 
without full page reloads. 
• Error messages are user-readable (e.g., 'Username must be 3-50 characters') rather than raw 
exception messages.

11. Tech Stack Justification 
 
11.1 Python Flask 
• Lightweight and minimal: Flask provides routing, request handling, and extension integration 
without imposing an MVC structure — ideal for academic projects where the team controls the 
architecture. 
• Blueprint support enables modular, scalable application organization without migrating to a 
heavier framework like Django. 
• Python's native integration with ML libraries (scikit-learn, TensorFlow, Surprise) makes Flask the 
natural choice for projects that will incorporate pre-trained models. 
• Alternative considered: Django — rejected due to ORM overhead and template-centric design 
that conflicts with the JSON API architecture. 
 
11.2 MySQL 
• Relational data model is ideal for CineVault's structured, relationship-heavy domain: movies 
have many genres, users have many reviews, reviews have many likes. 
• InnoDB's foreign key constraints and CASCADE rules enforce data integrity at the database 
layer — not just the application layer. 
• GROUP_CONCAT and window functions enable efficient aggregation (genre lists, trending 
scores, collaborative filtering) without multiple round-trips. 
• Alternative considered: PostgreSQL — equivalent capability; MySQL was chosen for its ubiquity 
in academic environments and WAMP/XAMPP local setups. 
 
11.3 Vanilla JavaScript 
• Zero build toolchain: no npm, webpack, or transpilation needed. Students can run the project 
with just Python installed.

12. Current Limitations 
 
12.1 Dataset Limitations 
• The seed dataset contains only 10 movies across 10 entries in schema.sql. In production, the 
platform requires integration with a movie metadata API (TMDb, OMDb) or a larger pre-loaded 
dataset (e.g., MovieLens 25M). 
• Genre taxonomy is limited to 12 predefined genres. Niche genres (Anime, Documentary, Short 
Film) are absent. 
• Metascore and box office data are manually entered strings — not dynamically fetched from an 
authoritative source. 
 
12.2 Recommendation System Limitations 
• The collaborative filtering algorithm requires a minimum critical mass of reviews across users. 
With a small dataset or few active users, the self-join returns empty results and falls back to top-
rated. 
• Content-based filtering uses only genre as the feature vector. A richer model would incorporate 
director, cast, plot keywords (TF-IDF), language, and runtime. 
• No matrix factorization or embedding-based approaches are implemented in v2 — these require 
pre-trained model files and inference infrastructure. 
• Recommendation results are not cached — each request re-executes the multi-table JOIN 
queries.

12.3 System Limitations 
• No email verification for new user registration — the email field is stored but not validated via a 
verification link. 
• Session-based authentication does not support JWT tokens — unsuitable for mobile app or 
multi-tab scenarios where cookie sharing may be inconsistent. 
• The trending score is a simple incrementing counter — it does not decay over time. A 
production system would implement a time-decay function (e.g., Reddit's decay algorithm). 
• No rate limiting is implemented on the API endpoints — a DoS attack could overwhelm the 
MySQL connection pool. 
• Image/poster data relies on external URLs (Google CDN, TMDb). If external URLs break, poster 
images will fail to load.

15. Conclusion 
 
15.1 Project Impact 
CineVault successfully demonstrates that a production-grade, feature-rich web application can be built 
on a minimal, well-structured technology stack. The project delivers real value across five distinct feature 
domains — authentication, movie discovery, reviews, watchlist management, and intelligent 
recommendations — all integrated into a cohesive user experience. 
 
The multi-strategy recommendation engine is the most technically sophisticated component of the 
project. By implementing content-based filtering, user-based collaborative filtering, and a cold-start 
handler — all within SQL queries orchestrated by a smart routing layer — CineVault demonstrates an 
understanding of recommendation system design that goes beyond the typical final-year project scope. 
 
15.2 Technical Learning Outcomes 
• Relational Database Design: 11-table normalized schema with junction tables, ENUM columns, 
composite primary keys, CASCADE rules, and analytical tables (trending, movie_views). 
• RESTful API Architecture: Blueprint-based Flask application with consistent HTTP status codes, 
parameterized SQL, and session-based authentication. 
• Recommendation Systems: practical implementation of content-based filtering, user-based 
collaborative filtering, and cold-start handling — with a smart routing decision layer. 
• Security Engineering: bcrypt password hashing, HttpOnly/SameSite cookies, SQL injection 
prevention via parameterized queries, and input validation with whitelist patterns. 
• Software Architecture: Application Factory pattern, separation of concerns via blueprints, 
extension isolation to prevent circular imports, and configuration centralization.
